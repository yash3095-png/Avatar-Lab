Avatar Lab - Project Overview

The Avatar Lab project is a comprehensive full-stack application designed to generate AI-driven avatar videos from text and images. It provides multiple ways to create avatars, including using open-source models (SadTalker and StyleTTS2) and integrating with the commercial HeyGen API.

The project follows a modern microservices-style architecture, split into three distinct, decoupled components:
1. Frontend Client (/client)
2. Backend Server (/server)
3. AI Model Server (/model-server)

Below is a detailed, section-wise classification of the project's architecture, tech stack, and core functionalities.

---

1. Frontend Architecture (/client)

The frontend is a highly interactive, animated user interface built to provide a seamless experience for creating and managing AI avatars.

Technical Stack
* Framework: React 19 with Vite and TypeScript.
* Styling: Tailwind CSS.
* Animations: Framer Motion and GSAP (GreenSock Animation Platform).
* UI Components: Radix UI (accessible, unstyled components).
* 3D Capabilities: @splinetool/react-spline for rendering 3D scenes (likely for the homepage or loading screens).
* Routing: React Router DOM.

Core Pages & Features
* / (Home): The landing page, likely featuring high-end animations and 3D Spline elements.
* /login & /register: Authentication pages for user access.
* /dashboard: The post-login dashboard where users manage their account and start new projects.
* /create: The interface for generating custom avatars using the open-source pipeline (SadTalker/StyleTTS2). Users likely upload an image, enter text, and initiate generation here.
* /heygen: A dedicated interface for generating avatars using the commercial HeyGen API instead of the local AI models.
* /history: A page displaying previously generated avatar videos, fetched from the backend database.

---

2. Backend Server (/server)

The backend server acts as the central coordinator. It manages users, handles database operations, proxies requests to third-party APIs (like HeyGen), and orchestrates requests to the Python AI Model Server.

Technical Stack
* Runtime: Node.js with Express.js and TypeScript.
* Database: MongoDB, interacted with via Mongoose.
* Authentication: JSON Web Tokens (JWT) and bcryptjs for secure password hashing.
* File Handling: multer for receiving uploaded images and audio files from the frontend.

Core Functionalities & Routes
* Auth Routes (/api/auth): Handles user registration, login, and token generation.
* Avatar Routes (/api/avatar): Manages metadata for generated avatars. It saves details about generated videos to the database and retrieves user history.
* HeyGen Proxy (/api/heygen): Acts as a secure proxy to interact with the HeyGen API. This ensures the HeyGen API key remains hidden on the server and isn't exposed to the client browser.
* AI Avatar Generation Orchestrator (/api/ai-avatar): This acts as a bridge to the Python Model Server. When a user requests a custom avatar, this route receives the payload, optionally does preprocessing, and forwards the heavy lifting to the Python server.
* Static File Serving: Serves the generated media (from final_videos and temp_audio_outputs) directly to the frontend.

---

3. AI Model Server (/model-server)

The Model Server handles the computationally expensive tasks: Text-to-Speech (TTS) and Image Animation.

Technical Stack
* Framework: FastAPI running on Python via Uvicorn.
* Models Included:
    * StyleTTS2: Used to convert user-provided text into highly realistic synthesized voice audio.
    * SadTalker: Used to take a static image (a face) and the synthesized audio, animating the face to match the spoken words.
    * GFPGAN: Used alongside SadTalker to restore and enhance the quality of the generated face.

Core API Endpoints
* POST /synthesize_voice/:
    * Takes text and a reference speaker wav file.
    * Calls StyleTTS2 to generate synthesized speech.
    * Saves a temporary .wav file into temp_audio_outputs.
    * Returns the path to this audio file.
* POST /generate_avatar_video/:
    * Takes a source image and the path to the previously generated audio file.
    * Sends these inputs to SadTalker.
    * Produces the final talking-head video.
    * Saves the result to final_videos and returns the URL.
* GET /temp_audio_playback: Allows the frontend to preview the generated audio before committing to full video generation.

---

Summary of the Data Flow (Custom Avatar Generation)

1. User Input: The user uploads an image and enters text on the frontend (/create).
2. Voice Synthesis: The frontend requests voice synthesis. The Node.js server proxies this to the Python model-server (/synthesize_voice/).
3. Audio Generation: StyleTTS2 generates the audio. The Python server returns the audio path.
4. Video Generation Request: The frontend then requests full video generation, sending the image and audio path.
5. Avatar Animation: The Python server runs SadTalker (/generate_avatar_video/), creating the final mp4.
6. Storage & Delivery: The video metadata is saved to MongoDB via the Node.js server, and the video itself is served statically back to the frontend to be displayed and added to the user's history.
